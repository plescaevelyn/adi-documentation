    A2B Commands

    The purpose of the A2B shell commands are to allow the users to interact with the A2B specific features of the Audio Starter as they relate to network discovery, modes of operation, etc. For audio specific routing of A2B though, please refer to the Audio Shell Commands. The sections below describe the A2B networking commands in more detail.

    Network Discovery With Error Handling (discover)

    Description

    This shell command will attempt to discover an A2B connected network using a commandlist file exported from Sigma Studio for any of the AD24x transceivers. It will also override any TDM settings in the commandlist file if settings do not match the system configuration, ensuring that no audio corruption/distortion will occur due to TDM setting mismatch. Outside of using a direction connection via USBi to Sigma Studio, an exported commandlist is the only way for the Audio Starter to discover a network. This allows greater flexibility with the Audio Starter, such that, many different A2B Network configurations can be used with a single Audio Starter platform. For more information regarding commandlist files, using Sigma Studio and exporting these files, refer to Exporting Command List Files and A2B Software

    ** Syntax **

    discover <a2b.xml> <verbose> <i2c_port> <i2c_addr>

    Inputs

    a2b.xml - This is the exported commandlist file that comes from SigmaStudio. This file contains the sequence of I2C commands (Write/Read/Delay) to be programmed for discovering and configuring the A2B transceiver nodes and peripherals as per the drawn schematic. Additionally, for files that reside in the non-default file-system, the drive specifier must be appended to the filename during configuration. If the discover command cannot find or open the file, an error will be returned. See the shell command drive for more details. Range(string, lower-case, optional, default = a2b.xml): <filename>.xml, where filename can be any string. verbose - This specifies the verbosity of the output, which can be useful for debugging error conditions. Range(integer, optional, default = 1): [0, 2], where 0 means none, or that the results will not print and the function will exit quietly, 1 means stdout, or that the results will be printed to the shell, 2 means syslog, and the results will be printed to the system log. i2c_port - This is the physical I2C Port that the A2B Transceiver is connected to. On the Audio Starter, DSPs, this is typically denoted as TWIn, where n is the port number. When using a physical port other than the default, refer to the hardware schematic to understand the proper port to use. Range(integer, optional, default = Default A2B TWI Port): [0, 5] i2c_addr - This is the physical I2C address of the A2B transceiver. Note that transceivers may be programmed by hardware or jumpers to set the I2C address so it is important to refer to the hardware schematic and A2B reference manual to better understand what the intended I2C address is. Range(integer, hexadecimal, optional, default = 0x68): Any 7-bit address

    Outputs

    Discovery Result - SUCCESS - Indicates that the discovery was successful WIRE OPEN OR WRONG PORT - Indicates that there is no physical wired connection to the transceiver in the network. This can happen if the wire is disconnected, or connected to the wrong port (main vs. sub-node port). I2C WRITE ERROR - This is an indication that the I2C address requested didn’t receive a response from the transceiver. This usually means the wrong I2C address has been used. SUCCESS with WARNING - This is an indication that the TDM configuration in the commandlist didn’t match the system configuration and was overriden. Note that the entire I2C write log is reported in the system log which can be retrieved by running the shell command *syslog (see above). You do not need to enable verbosity for logging. This can be useful for tracing the above errors.*

    Example Usage

    discover <file.xml>

    Network Discovery without Error Handling (cmdlist)

    Description

    This shell command will attempt to discover an A2B connected network using a commandlist file exported from Sigma Studio for any of the AD24x transceivers. Unlike the shell command discover, the command will not override any incorrect settings and will simply send the I2C configurations to the transceiver. This command also uses exported commandlist files from SigmaStudio. Refer to discovery for more details. This shell command should be used with caution!

    ** Syntax **

    cmdlist <cmdlist.xml> <verbose> <i2c_port>

    Inputs

    cmdlist.xml - - This is the exported commandlist file that comes from SigmaStudio. This file contains the sequence of I2C commands (Write/Read/Delay) to be programmed for discovering and configuring the A2B transceiver nodes and peripherals as per the drawn schematic. Additionally, for files that reside in the non-default file-system, the drive specifier must be appended to the filename during configuration. If the discover command cannot find or open the file, an error will be returned. See the shell command drive for more details. Range(string, lower-case, optional, default = cmdlist.xml): <filename>.xml, where filename can be any string. verbose - Unlike the discover command, verbosity only applies to errors (minor). If this command is successful, any success status, no matter the verbosity, will exit quietly. Range(integer, optional, default = 1): [0, 2], where 0 means none, or that the results will not print and the function will exit quietly, 1 means stdout, or that the results will be printed to the shell, 2 means syslog, and the results will be printed to the system log. i2c_port - This is the physical I2C Port that the A2B Transceiver is connected to. On the Audio Starter, DSPs, this is typically denoted as TWIn, where n is the port number. When using a physical port other than the default, refer to the hardware schematic to understand the proper port to use. Range(integer, optional, default = Default A2B TWI Port): [0, 5]

    Outputs

    Runs commands from xml file generated by SigmaStudio project to configure A2B network

    Example Usage

    cmdlist adi_a2b_commandlist_1.xml 1 0

    A2B Operating Parameters (a2b)

    Description

    The Audio Starters are capable of running as an A2B main node or an A2B sub node. This is useful if you want to connect more than one Audio Starter together via A2B. All of the Audio Starters start up in A2B main node mode so if a user wants to run an Audio Starter in sub node mode, this command must be used to change the system mode. When the mode is changed from main to sub, the Audio Starter will update it’s internal clocking scheme to allow the A2B to be clocked from a different external clock provided upstream by an A2B main node. This change in clocking can affect other system audio streams, such as codec and wav file streaming. As such some audio streams may need to be manually updated (in software) or by a configuration, where possible, to run on the A2B clock domain rather than the system to be able to stream audio from the A2B bus coming into the sub node and into another audio sink within the system. Additionally, some Audio Starters run on hardware which can support more than one main/sub node. In these systems, these are denoted as a2b and a2b2 and by default, have I2C addresses as 0x68 and 0x6A, respectively. This command allows users to also change these default settings.

    Inputs

    cmd - This is the a2b command whose parameter should be modified. Range(string, lower-case): [mode, addr]. The input parameters vary depend on the command used: mode - This is the A2B operating mode of the system to set. Range(string, lower-case, optional, default = main): [main, sub] addr - This is the I2C address to set for a2b or a2b2. Parameter #1 - Range(string, lower-case, optional): [a2b, a2b2] Parameter #2 - Range(integer, hexadecimal, optional, default = [a2b (0x68), a2b2 (0x6A)]): Any 7-bit address

    Outputs

    Without any inputs (above) - the current A2B parameters of the system will be displayed.

    ** Syntax **

    a2b

    a2b cmd

    Example Usage

    a2b or a2b mode main The following updates I2C address (where supported on applicable hardware) from their default settings: a2b addr a2b 0x6A and a2b addr a2b2 0x68

    Expansion Port AD242x (tdm)

    Note

    Support status:

    • ADZS-SC589-MINI - TBD

    • ADZS-SC584-EZLITE - TBD

    • EV-SC594-SOM - TBD

    • EV-SC598-SOM - TBD

    Description

    This command allows the user to configure the TDM interface of an AD242x device connected to the SAM board expansion port. Expansion port pin assignments: - TX0: P5.1 - TX1: P5.3 - BCLK: P5.5 - SYNC: P5.9 - RX0: P5.11 - RX1: P5.13

    ** Syntax **

    // tdm [cmd] [args] // // tdm enable // // tdm disable // // tdm reset // // tdm dir <in

    Inputs

    GENERAL cmd - This selects the specific TDM subcommand to execute. Range (string, required): [enable, disable, reset, dir, fs, I2SCFG, I2SGCFG] COMMAND SPECIFIC in|out - This sets the TDM port direction from SAM perspective. // Range (string, lower-case, optional, default = out): [in, out] // fs - This sets the TDM port sample rate. // Range (integer, optional, default = 48000): [0,] // I2SCFG - This sets the external port AD242x’s I2S Configuration Register value. See AD242x Technical Reference for more information about this register. // Range (hex, 8 bit, required, default = 0x19): [0x00, 0xFF] // I2SGCFG - This sets the external port AD242x’s I2S Global Configuration Register value. See AD242x Technical Reference for more information about this register. // Range (hex, 8 bit, required, default = 0xE0): [0x00, 0xFF] //

    Outputs

    TDM Settings

    Example Usage

    // tdm // This will display simple TDM settings. // tdm -v // This will display verbose TDM settings. // tdm dir in // // tdm i2sgcfg 0xe0 // // tdm i2scfg 0x19 // // tdm enable // This will configure the TDM port as input and enables Invert Sync, Early Sync, Alternate Sync, RX0, and TX1 before enabling the port.
