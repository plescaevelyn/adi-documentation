    Development and Debug Commands

    The purpose of these commands is to help users or developers of the Audio Starter debug system level issues, issues with hardware connections, etc. This section describes each of the development and debug related shell commands in detail.

    Live File Editing (edit)

    Description

    This command allows you to edit files, live, in one of the supported file-systems on the Audio Starter. Additionally, this command will create a new file for editing, if the file does not currently exist. This is especially useful for creating new command files to run on target (See the command run) or for inspecting files for correctness. To edit or create a file for editing that resides in the non-default file-system, append the device drive identifier to the filename. See the command drive for more details.

    ** Syntax **

    edit <filename>

    Inputs

    filename - This is the filename to open or create for editing. Range(string, required): any

    Outputs

    Edit View of file

    Example Usage

    edit test.cmd This creates a new file called test.cmd in the default file-system and opens it for edit. The following commands can be used for editing: Ctrl-S - Control Key + s-key (lowercase) = Save Ctrl-Q - Control Key + q-key (lowercase) = Quit Ctrl-F - Control Key + f-key (lowercase) = Find Typing in a command like ver and then saving will display the number of bytes saved on the disk. Quitting will then close the file which allows it to be accessed later.

    Example Shell Command (test)

    Description

    This is an example shell command that users can use to experiment with the codebase further. All of the hooks for shell execution are already in place so if a user needs to execute some functionality quickly or just wants to try out compilation and dive deeper into the Audio Starter, this is a great starting point. Currently, this shell command is empty: But it could be populated with experimental or test code: Which will become visible when the shell command is executed (after re-compilation and re-flashing of course :))

    ** Syntax **

    test [args…]

    Inputs

    None

    Outputs

    None

    Example Usage

    test

    Automated On-Target Command Execution (run)

    Note

    Short options support status:

    • ADZS-SC589-MINI - v4.0.0 or greater

    • ADZS-SC584-EZLITE - TBD

    • EV-SC594-SOM - TBD

    • EV-SC598-SOM - TBD

    Description

    This command executes a series of shell commands that are stored in a .cmd that lives in one of the available file-systems on the Audio Starter. This can be useful for quickly and automatically running a series of commands in a sequence, especially as the state of the Audio Starter is not persistent across a power-cycle. For example, if a user is using the Audio Starter for a demo application that needs to set up a series of routes with a wav file enabled and streaming, it may be more convenient to simply save that series of commands to a file on the Audio Starter, rather than re-type them into the shell after each power cycle. Generally these type of files are denoted as *.cmd. Outside of this command, .cmd files can also be run via the pushbuttons (pushbtn1.cmd and pushbtn2.cmd in the default file-system) on the hardware and automatically at startup (file must be named shell.cmd and must reside in sf:). To run a command file that resides in the non-default file-system, append the device drive identifier to the filename. See the command drive for more details. Run command may be interrupted by pressing CTRL + C.

    ** Syntax **

    run [options] <file1> [<file2> …]

    Inputs

    file1 - This is the name of the .cmd file to execute. Range(string, required): any Note that subsequent files are optional. OPTIONS -l <loop count> This option can be used to run one or more command files the determined number of times. If not set, command files will be run once.

    Outputs

    Prints any outputs executed by commands that reside in the .cmd file.

    Example Usage

    run test.cmd This will run a file called test.cmd that resides within the default file-system. The commands that live within this file are visible with the cat command (see below). Each command is separated by a new line. run -l 10 file1.cmd file2.cmd This will run file1.cmd and file2.cmd 10 times.

    Internal Flash Hex Dump (dump)

    Description

    This command dumps the raw bytes of the internal flash file-system, starting from a requested address specified by the inputs.

    ** Syntax **

    dump [addr] <len>

    Inputs

    addr - This is the starting address to start dumping Range(long integer, required): [0 MAX_UL] len - This is the maximum number of bytes to dump Range(long integer, optional, default = 1): [0, MAX_UL] Both decimal and hexadecimal are accepted formats.

    Outputs

    Displays bytes from memory address

    Example Usage

    dump 10 6 This will dump 6 bytes starting from address 0xA.

    File-System File Hex Dump (fdump)

    Description

    This command dumps the raw bytes of a given file from a supported file-system, starting from a requested address specified by the inputs. To dump raw bytes of a file that resides in the non-default file-system, append the device drive identifier to the filename. See the command drive for more details.

    ** Syntax **

    fdump [file] <start> <size>

    Inputs

    file - This is the name of the file whose raw bytes are to be dumped. Range(string, required): any start - This is the starting address into the file whose contents are to be dumped. Range(long integer, optional, default = 0): [0, MAX_UL] size - This is the size, in bytes, to dump. Range(long integer, optional, default = Full File): [0, MAX_UL]

    Outputs

    Displays file content in hex format

    Example Usage

    fdump test.bin 0x20 32 This dumps the contents of the file test.bin starting at address 0x20 with 32 byes dumped.

    System Delay (delay)

    Description

    This invokes a system delay. This is useful adding timing sequences to .cmd files. See run for more details.

    ** Syntax **

    delay <ms>

    Inputs

    ms - The amount of time to delay, in milliseconds. Range(long integer, optional, default = 0 (no delay)): [0, MAX_UL]

    Outputs

    None

    Example Usage

    delay 1000 In the shell, this executes a delay of 1 second. An example of a delay in a .cmd which routes the ADC Input to a wave file for recording, turns on a wave sink, delays for 10 seconds to finish the recording, then turns the recording off.

    Execute Raw I2C Transfer (i2c)

    Description

    This command executes a raw I2C read or write transfer. This is useful for communicating directly with on-board I2C devices for debugging purposes (without the need for modifying direct source code). Note that this command can handle both 8-bit and 16-bit register addresses (mem_addr in the input list below), but only supports 7-bit addressing (10-bit addressing is NOT supported). At the lowest layer, this command calls a single writeRead API, which inserts a repeated start condition in-between the write and read transaction. For example, the below scenario shows a single read, with a 16-bit register address, reading two bytes of data: The following formats are supported: Single read with 8-bit or 16-bit memory addresses: Burst read with 8-bit or 16-bit memory addresses: Single write with 8-bit or 16-bit addresses: Burst write with 8-bit or 16-bit addresses: Note that the structure of the I2C read or write can vary from device to device. It is important to refer to the device’s datasheet to understand the proper formatting of the I2C message. And although this command attempts to be flexible enough to cover a range of formats, it is not guaranteed to work with every single I2C device format that exists.

    ** Syntax **

    i2c <i2c_port> <i2c_addr> <mem_addr> <wdata> <length> [addr_len]

    Inputs

    i2c_port - This is the physical I2C Port that the I2C device is connected to. Refer to the hardware schematic to understand the proper port to use. Range(integer, required) = [0, 5] i2c_addr - This is the physical I2C address of the I2C device. Range(integer, hexadecimal, required): Any 7-bit address mem_addr - This is the starting address in the device’s register address space for reading or writing to. Range(integer, required): [0, MAX_U16] wdata - This is the raw data that will be used to write to the I2C device, in comma-separated, string format. Range(string, comma-separated, required): Integer or hexadecimal values enclosed by double-quotes. Use empty quotes for a read-only operation. length - The number of bytes to read. Range(integer, required): [0, MAX_UL] addr_len - The size, in bytes, of the memory address. Range(integer, optional, default = 1): [1, 2], 1 = 1-byte memory address, 2 = 2-byte memory address The command will try to infer this parameter value by checking the size of the i2c_addr value. If this value is greater than 255, the default memory address size will be 2-bytes rather than 1-byte. For devices which require an upper and lower byte, even in the range less than 255, the user MUST still input 2 for this parameter.

    Outputs

    On Reads I2C Device - This is the 7-bit address that was read from addr - This is the starting address that was read from bytes read - This is the number of bytes that was read, in both decimal and hexadecimal format n-byte address length - This is the address length format that was used during the read operation <addr>: <data> - This is the actual data read

    Example Usage

    i2c 0 0x38 0x4000 “” 1 2 This reads one byte from register address 0x4000 from device 0x38 (ADAU1761) i2c 0 0x38 0x4002 “” 6 2 This reads 6 bytes starting from register address 0x4002 from device 0x38 (ADAU1761) i2c 0 0x38 0x4000 “0x03” 0 2 and i2c 0 0x38 0x4000 “” 1 2 This writes one byte of data (0x03) to register 0x4000 to device 0x38 and then proceeds to read the same register to verify the same contents were read back.

    Check for Active I2C Devices (i2c_probe)

    Description

    This command looks for active external ICs connected via I2C by running through the entire 7-bit I2C address space. Note that this command only works when the physical I2C port is configured in the project. That is, the I2C peripheral itself is opened and configured and any physical hardware pin-muxing is already set up for I2C functionality.

    ** Syntax **

    i2c_probe <i2c_port>

    Inputs

    i2c_port - This is the physical I2C Port to probe. On the Audio Starter running DSPs, this is typically denoted as TWIn, where n is the port number. Refer to the hardware schematic to understand the proper port to use. Range(integer, required): [0, 5]

    Outputs

    Displays I2C device addresses for entered I2C port

    Example Usage

    i2c_probe 0

    Stack Usage (stacks)

    Description

    This returns the number of words remaining in the task stack. For example, on a 32 bit machine a return value of 1 would indicate that 4 bytes of stack were unused. If the return value is zero then the task has likely overflowed its stack. If the return value is close to zero then the task has come close to overflowing its stack. Stack usage should be checked periodically during development to ensure usage is not approaching an overflow condition. Note that the stack here is referring to the tasks allocated to an OS task rather than the processor stack. See Stack Usage for more details.

    ** Syntax **

    stacks

    Inputs

    None

    Outputs

    Number of words remaining in stack, per configured task

    Example Usage

    stacks

    CPU System Utilization (cpu)

    Description

    This command reports the processor utilization. On the ARM, this value is a percentage, whose value is inversely proportional to the amount of time spent idling. That is, the higher the percentage, the less time the processor is spent idling, indicating that more time is being spent executing in ISR context. A higher value or values approaching 100% indicates that the system is running out of cycles to execute all processes on time. On the SHARC processors, this value is reported in CYCLES and is the measurement of cycles spent processing SHARC input and output audio. Note that the peak usage value reported for the ARM is cleared once this command is called. That is, the next peak value reported is the peak value calculated since the last time this command was called and is not the value reported of all time.

    ** Syntax **

    cpu

    Inputs

    None

    Outputs

    Displays cpu usage of all cores with current and peak usage

    Example Usage

    cpu

    System Logging Report (syslog)

    Description

    This command reports the various internal logging events of the system that may be of use for users of the Audio Starter, such as SD Card connection status, ethernet status, etc. This command will remain open and active to report new events as they appear until the user presses enter. Note that once the user presses enter, thus exiting the live system log, all of the previously displayed system log contents are flushed. This means that the next time the syslog command is executed, only the latest events logged will be displayed. If nothing has been logged, it will simply display an empty system log.

    ** Syntax **

    syslog

    Inputs

    None

    Outputs

    Displays live system log

    Example Usage

    syslog

    USB Audio Statistics (usb)

    Description

    This command displays useful statistics related to USB (UAC2) audio. It also allows you to change the UAC2 audio clock domain to/from A2B and System domains as well as reset the statistics. This can be useful while debugging USB audio related tasks or issues.

    ** Syntax **

    usb

    Inputs

    in|out This is the direction (RX/TX) of USB whose domain will be changed or whose statistics will be reset. Range(string, lowercase, required): [in, out] domain - This is the clock domain that the USB (UAC2) audio should be running on. Range(string, lowercase, required): [a2b, system] reset - This clears the statistics. Requires the in|out parameter when used. Range(string, lowercase): [reset]

    Outputs

    Displays data about Rx and Tx USB packets

    Example Usage

    usb usb out reset This clears the statistics for USB out (Rx). usb in domain a2b This sets the domain of the USB in (Tx) to be on the A2B Domain.

    Dynamic Memory Allocation Usage (meminfo)

    Description

    This command displays the usage of all of the dynamically allocated memory in the system. For heaps managed by umm_malloc, each heap type, the name of the heap, the total heap size, the total amount of heap allocated and the remaining amount of heap left (units are in bytes) is displayed. Additionally, for heaps managed by umm_malloc, the integrity of the memory is also verified. This can be a useful diagnostic when checking for buffer overflows and other areas of memory corruption during development. Note that heap management is split between system (*umm_malloc), FreeRTOS (heap_4) and SAE (internally managed).*

    ** Syntax **

    meminfo

    Inputs

    None

    Outputs

    **Heap <heap_name> Info:** Total = Total assigned in memory, Allocated = Memory/entry currently in-use, Free = Memory/entry no longer allocated Entries - This refers to the number of buffers that were allocated for this heap. Blocks - This refers to the physical memory, in bytes, for total heap. Contig - This refers to the physical contiguous memory, in bytes, for total heap, accounting for alignment rules. Heap Integrity - This specifies if the heap is ok to use or if it has any corruption which should be investigated further. FreeRTOS Info: Free: Blocks - This is the number of blocks remaining in the FreeRTOS heap. Current - This is the available heap space, in bytes, remaining. Min - This was the minimum number of bytes that were ever free. SHARC Audio Engine Info: Blocks: Total - This is the total blocks available in the SAE heap. Allocated - This is the number of blocks currently allocated. Free - This is the number of free blocks remaining. Size: Free - This is the size of the free heap, in bytes. Allocated - This is the amount of memory, in bytes, allocated by the SAE. Max - This is the maximum size of the contiguous free heap, in bytes.

    Example Usage

    meminfo

    Software Reflash (update)

    Description

    This command provides the ability to update the firmware on the external flash from a user’s PC (via XMODEM) and put them onto one of the available file-systems on the Audio Starter. Note that the user MUST use a terminal program that supports XMODEM transfers to use this command. See the Shell Overview for more details. Users must perform a system reset or a power-on reset for the changes to take effect. Warning! There is no validity check on the file being transferred so use this command with caution! Failure to do so can cause your external flash to become corrupted and not bootable, requiring you to recover via a different supported flashing method for your platform. See Software Update for details.

    ** Syntax **

    update

    Inputs

    Firmware to be updated via xmodem in .LDR format

    Outputs

    None

    Example Usage

    update

    Shell Screen Resize (resize)

    Description

    This command provides the ability to resize the shell terminal screen via command. The size specified is in the size of columns, line, where each column and line is equivalent to the size of a single character displayed on the screen. Note that an input of columns = 0, lines = 0 resets the terminal to a standard size.

    ** Syntax **

    resize [columns [lines]]

    Inputs

    columns - This is the number of characters to size the columns of the terminal Range(integer, optional): [0, MAX_UL] lines - This is the number of characters to size the lines of the terminal Range(integer, optional): [0, MAX_UL]

    Outputs

    Current columns and lines, in character width

    Example Usage

    resize 20 40 This sets the column size to 20 characters and the lines size to 20 characters. resize This displays the current size of the terminal.

    Shell Output Redirect (shell)

    Note

    Support status:

    • ADZS-SC589-MINI - v4.0.0 or greater

    • ADZS-SC584-EZLITE - TBD

    • EV-SC594-SOM - TBD

    • EV-SC598-SOM - TBD

    Description

    This command provides the ability to redirect print statements from the system and shell commands to something other than standard output, i.e. to the system log or to a specified output file.

    ** Syntax **

    shell [redirect] [value] [arg]

    Inputs

    redirect - This is the action that the shell command should execute. Note that this may be expanded in the future. Range(string, optional): [redirect] value - This is the value associated with the where system printing should route to A value of 0 selects none which will completely suppresses any print statements. A value of 1 will select stdout which will print directly to standard out (i.e. shell, telnet, etc.). A value of 2 selects syslog which re-routes system printing to the system log. See syslog for more details about system logging. A value of 3 selects file which re-routes system printing to a file, specified by the user. Range(integer, optional, default = 1 (stdout)): [0, 1, 2, 3] arg - This input value is ONLY used when the redirect value is set to 3:file. Use this parameter to set the filename of your choosing for the printed output. Range(string, required (when *value = 3): any*

    Outputs

    None

    Example Usage

    shell redirect or shell redirect 1 This sets the system printing to standard out. shell redirect 0 This completely suppresses system printing. shell redirect 2 This re-routes system printing to the system log. shell redirect 3 printFile.txt This re-routes system printing to a requested file.

    Argument Echo (echo)

    Note

    Support status:

    Description

    This command provides the ability to echo a string back to the system output. This can be useful for debugging and scripting of the shell command file (.cmd). See the run command for more details. No quotes, single quotes and double quotes are all accepted.

    ** Syntax **

    echo [arg_1] … [arg_n]

    Inputs

    arg_1 - arg_n - String argument to be displayed on the system out. Range(string, optional): any

    Outputs

    Displays arg_1 - arg_n

    Example Usage

    echo or echo Hello World!
